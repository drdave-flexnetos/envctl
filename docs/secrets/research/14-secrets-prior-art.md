# env-ctl research — Prior art: secrets brokers + machine identity

> Research date: 2026-06-02. Every nontrivial claim below carries an inline source URL. Items the author could not verify against a live page are explicitly flagged with ⚠ **UNVERIFIED** or ⚠ **CORRECTED**.

---

## TL;DR — recommendation for env-ctl

env-ctl's core design (local-+server vault, app-layer XChaCha20-Poly1305, argon2id keyslots, USB-UUID unlock, gRPC-over-UDS control plane, HTTPS MITM relay with local CA, ≤24h USB-gated relay bearers) is **architecturally aligned with the 2025–2026 industry consensus** for secrets brokering and machine identity. Concretely:

1. **The "virtual-credit-card relay" model is now the mainstream pattern, not a novelty.** Infisical Agent Vault (GA April 2026) and 1Password Unified Access (GA March 2026) both ship exactly this: a local TLS-intercepting forward proxy with a trusted local CA that strips agent-supplied auth and injects real upstream credentials, so the agent never sees the secret. env-ctl should position itself in this category and borrow their threat framing. ([Infisical](https://infisical.com/blog/agent-vault-the-open-source-credential-proxy-and-vault-for-agents), [1Password](https://1password.com/press/2026/mar/1password-unified-access))
2. **Short-lived, audience-scoped, auto-renewed credentials are the universal short-list of best practices.** Vault response-wrapping (single-use cubbyhole tokens), Vault dynamic secrets (per-request unique creds + lease revocation), Sigstore Fulcio (10-minute certs, revocation unnecessary), SPIFFE/SPIRE SVIDs (renew at 50% TTL), and RFC 9068 (`aud` claim mandatory) all converge here. env-ctl's ≤24h bearer is conservative-but-fine; pair it with a per-resource audience binding and resolve-once peer scoping.
3. **For the libSQL store: decide the C-dependency question explicitly.** Mainline **libSQL bundles the C SQLite amalgamation** (`libsql-ffi` compiles `sqlite3.c` via `cc`/`cmake`/`bindgen`). If env-ctl wants a *pure-Rust* engine, that requires the **Turso** rewrite (SQLite-compatible, in-process, native per-page encryption GA Oct 2025), not libSQL. This is a real fork-in-the-road, not a footnote. ([libsql-ffi](https://github.com/tursodatabase/libsql), [Turso](https://turso.tech/blog/introducing-fast-native-encryption-in-turso-database))

**Net:** keep the design. Tighten three things — audience-scoped bearers, a documented C-deps ruling for the store, and explicit lease/revocation semantics modeled on Vault.

## Key facts (with inline source URLs)

### Credential brokers / relay-proxy model

- **Infisical Agent Vault** — open-source, local **TLS-intercepting credential-injecting forward proxy** for AI agents. Using a **locally trusted CA**, it presents itself as the upstream service, lets the agent connect and send the request in plaintext over the tunnel, then **strips agent-supplied credentials, injects the correct credential from its encrypted store, and opens a fresh verified TLS connection to the real upstream**. The agent is unaware the proxy exists ("no surface to reason about, probe, or circumvent"). Written in **Go**. ⚠ **CORRECTED:** GA date is **April 22, 2026** (the upstream task brief's "research preview" framing is stale). ([Infisical blog](https://infisical.com/blog/agent-vault-the-open-source-credential-proxy-and-vault-for-agents), [Infisical explainer](https://infisical.com/blog/credential-brokering-for-ai-agents), [GitHub](https://github.com/Infisical/agent-vault))
- **1Password Unified Access** — GA **March 17, 2026** (Unified Access Pro generally available at launch). Three pillars: **Discover** (find agents + exposed creds/SSH keys), **Secure** (govern human/agent/machine creds in a unified vault — available now), **Audit** ("coming soon"). Launch partners: Anthropic, Cursor, GitHub, Perplexity, Vercel. ⚠ **CORRECTED:** **Runtime scoped credentials for agent/machine workloads are "later this year" (planned), not yet deployed** — do not claim this feature is shipping. ([1Password press release](https://1password.com/press/2026/mar/1password-unified-access))
- **HashiCorp Vault — response wrapping.** Vault inserts the response into the **cubbyhole of a single-use wrapping token** and returns that token instead of the secret. Unwrap is a one-time operation against the token; after unwrap (or TTL expiry/revoke) **both the token and its cubbyhole are destroyed**. The secret never appears in transit/logs, and interception is detectable (the legit recipient's unwrap fails). ([Vault response wrapping](https://developer.hashicorp.com/vault/docs/concepts/response-wrapping))
- **HashiCorp Vault — dynamic secrets + leases.** Credentials are **created just-in-time per request** (e.g., a unique DB user + random password per call), not stored. Each read returns a **`lease_id`** with a TTL; leases can be renewed (increment from *now*, not from current end) or revoked. **Revoke invalidates the secret immediately and blocks further renewal; expiry triggers automatic revocation.** ([Vault lease docs](https://developer.hashicorp.com/vault/docs/concepts/lease), [Vault dynamic secrets](https://developer.hashicorp.com/vault/tutorials/get-started/understand-static-dynamic-secrets))
- **Doppler — rotation patterns.** Issuer rotation (create new + revoke old, maintaining an active/inactive pair) vs. updater rotation (modify existing in place); per-consumer API-key scoping with permission matching. ⚠ **UNVERIFIED (this pass):** the specific Doppler rotation doc URL was not re-fetched in this session; the pattern itself is corroborated by the Vault/Infisical sources above. Treat the Doppler-specific citation as carried over from prior research, not freshly confirmed. ([Doppler rotation docs](https://docs.doppler.com/docs/supporting-rotation-in-your-product))

### Machine identity / ephemeral certs

- **SPIFFE / SPIRE — SVID renewal.** ⚠ **CORRECTED (was 80%):** SPIRE Agent **proactively renews SVIDs at 50% of their TTL** (half-life), e.g. a 1-hour cert refreshes ~every 30 min; a 7-day TTL refreshes after 3.5 days. The agent requests new X.509-SVIDs only for cached entries past the half-life of their TTL. (Jitter to de-spike renewals is a discussed enhancement, not the default.) ([SPIRE issue #4268](https://github.com/spiffe/spire/issues/4268), [SPIRE issue #1447](https://github.com/spiffe/spire/issues/1447), [SPIFFE overview](https://sameerbhanushali.substack.com/p/spiffe-and-spire-the-workload-identity))
- **Teleport Machine ID (`tbot`).** ⚠ **CORRECTED (was "GA Feb 2025"):** Machine ID was **introduced GA in Teleport 9, released April 5, 2022.** `tbot` renews on a configured period and initiates renewal in **four scenarios**: a fraction of the cert TTL elapsed/nearing expiry, user/host CA rotation, manual request, and `tbot` startup. It monitors the CAs and triggers extra renewals on CA rotation so outputs always carry certs from the latest CA. ([Teleport "How we built Machine ID"](https://goteleport.com/blog/how-we-built-machine-id/), [tbot deploy docs](https://goteleport.com/docs/machine-workload-identity/deployment/))
- **Sigstore Fulcio — keyless ephemeral certs.** Fulcio is a free code-signing CA issuing **X.509 certs valid for 10 minutes**, bound to an OIDC identity (subject claims embedded in the SAN). Lifespan is deliberately short enough that **revocation management is unnecessary**; the signing private key is ephemeral, in-memory only, never on disk, never known to Sigstore services. ([Fulcio overview](https://docs.sigstore.dev/certificate_authority/overview/), [OIDC in Fulcio](https://docs.sigstore.dev/certificate_authority/oidc-in-fulcio/))
- **Sigstore Rekor v2 — transparency log.** Backed by a **tile-based log on Trillian-Tessera**; cheaper to run, CDN-cacheable reads, single endpoint `/api/v2/log/entries`, with **witnessing integrated directly into Rekor** for stronger append-only guarantees. ⚠ **NUANCE:** As of **October 2025** the Rekor v2 log key began TUF rollout, but **the v2 URL was not yet distributed in `SigningConfig`** — planned for **end-2025/early-2026** to give clients time to upgrade. So "GA" is the code/log; default client cutover lagged. ([Rekor v2 GA blog](https://blog.sigstore.dev/rekor-v2-ga/), [rekor-tiles](https://github.com/sigstore/rekor-tiles))

### Token / API scoping standards

- **RFC 9068 (JWT Profile for OAuth 2.0 Access Tokens).** The **`aud` (audience) claim is REQUIRED**; each resource server **MUST reject** a token whose `aud` doesn't list it. Rationale: without audience validation, a token minted for API A can be **replayed against API B** behind the same authorization server. If `resource` is absent, the AS MUST use a default resource indicator (inferring from `scope` when present). ([RFC 9068](https://datatracker.ietf.org/doc/html/rfc9068))

### Storage / crypto primitives

- **libSQL (mainline) is NOT pure Rust.** `libsql-ffi` compiles the **SQLite C amalgamation (`sqlite3.c`)** plus optional encryption libs via `cc`/`cmake`, then generates Rust bindings with `bindgen`; encryption uses **SQLite3 Multiple Ciphers** bundled as C. ([libSQL GitHub](https://github.com/tursodatabase/libsql), [libsql-ffi on docs.rs](https://docs.rs/libsql-ffi/latest/libsql_ffi/))
- **Turso (the SQLite-in-Rust rewrite) IS the pure-Rust path.** In-process, SQLite-compatible. **Native encryption GA October 2025**: per-page (4 KiB) encryption, ~**6% read / ~14% write overhead** on large DBs with AEGIS-256, negligible on small DBs. ([Turso encryption blog](https://turso.tech/blog/introducing-fast-native-encryption-in-turso-database), [Turso encryption docs](https://docs.turso.tech/tursodb/encryption), [Turso GitHub](https://github.com/tursodatabase/turso))
- **Argon2id (RFC 9106).** Recommended default KDF for password hashing — hybrid construction (Argon2i first half, Argon2d remainder) balancing side-channel and brute-force resistance. ([RFC 9106](https://datatracker.ietf.org/doc/rfc9106/)) ⚠ **UNVERIFIED (this pass):** RFC 9106 not re-fetched live in this session; widely-cited and stable, but treat as carried-over.
- **XChaCha20-Poly1305.** 192-bit (24-byte) nonce, stateless AEAD construction — random nonces are safe at scale, making it well-suited to per-record envelope encryption. ([libsodium XChaCha20-Poly1305](https://libsodium.gitbook.io/doc/secret-key_cryptography/aead/chacha20-poly1305/xchacha20-poly1305_construction)) ⚠ **UNVERIFIED (this pass):** libsodium page not re-fetched live; the 192-bit nonce / stateless property is well-established.

### gRPC over UDS peer auth

- **`SO_PEERCRED` is not built into gRPC for UDS.** The standard pattern is a custom `TransportCredentials` whose `ServerHandshake` extracts peer info from the `net.Conn`, wraps it in an `AuthInfo`, and exposes it via `peer.FromContext()`. In Rust this maps to reading `SO_PEERCRED` directly (e.g. `rustix` with the `process`/`net` features). ([gRPC issue #15511](https://github.com/grpc/grpc/issues/15511)) ⚠ **UNVERIFIED (this pass):** issue #15511 not re-fetched live; carried over from prior research.

## Security tradeoffs

- **Short TTL vs. revocation infrastructure.** Fulcio's 10-min certs eliminate revocation entirely; SPIRE's 50%-TTL renewal keeps creds fresh at the cost of ~2× the renewal traffic vs. a hypothetical 80% strategy. **env-ctl's ≤24h bearer sits at the long end** — acceptable because it's USB-gated and peer-bound, but it means env-ctl *does* need real revocation (you cannot rely on "too short to matter"). Budget for an explicit revoke path à la Vault leases. ([Fulcio](https://docs.sigstore.dev/certificate_authority/overview/), [SPIRE](https://github.com/spiffe/spire/issues/4268), [Vault lease](https://developer.hashicorp.com/vault/docs/concepts/lease))
- **MITM proxy = single high-value interception point.** The relay/local-CA pattern (Infisical, 1Password, env-ctl) concentrates *all* upstream secrets at the proxy. Benefit: the agent is blind to secrets and has "no surface to probe." Cost: the proxy's local CA private key and its credential store become the crown jewels; compromise is total. This is exactly why env-ctl's USB-gated DEK unlock and per-record AEAD matter — they keep the store useless without the physical key. ([Infisical](https://infisical.com/blog/agent-vault-the-open-source-credential-proxy-and-vault-for-agents))
- **Audience binding vs. broad bearers.** RFC 9068 makes the case bluntly: a bearer with no `aud` is replayable across any API behind the same issuer. env-ctl's relay bearers should carry a resource/audience binding so a leaked bearer can't be redirected at a different upstream. ([RFC 9068](https://datatracker.ietf.org/doc/html/rfc9068))
- **Per-request dynamic creds vs. injected static creds.** Vault's gold standard is *generating* a unique credential per consumer so revocation and audit are trivial. The relay model (Infisical/1Password/env-ctl) instead *injects a shared real credential* the agent never sees — simpler and works with upstreams that don't support dynamic issuance, but the underlying upstream secret is long-lived and shared, so **rotation discipline on the real secret still matters**. env-ctl gets the best of both only if it can rotate injected upstream secrets on a schedule. ([Vault dynamic secrets](https://developer.hashicorp.com/vault/tutorials/get-started/understand-static-dynamic-secrets), [Infisical](https://infisical.com/blog/credential-brokering-for-ai-agents))
- **Pure-Rust store vs. C-bundling SQLite.** libSQL's bundled C `sqlite3.c` is mature and battle-tested but expands the unsafe/C attack surface and complicates a "no-C" supply-chain story. Turso is pure Rust with native encryption but is a younger engine. This is a **security-vs-maturity** tradeoff, not just a build-system preference. ([libSQL](https://github.com/tursodatabase/libsql), [Turso](https://turso.tech/blog/introducing-fast-native-encryption-in-turso-database))

## Concrete guidance for the env-ctl implementation

1. **Add an audience/resource binding to every relay bearer.** Mirror RFC 9068: each ≤24h bearer should encode the specific upstream(s) it may be replayed against, and the relay must reject a bearer presented for the wrong audience. Cheap to add, kills cross-upstream replay. ([RFC 9068](https://datatracker.ietf.org/doc/html/rfc9068))
2. **Model the bearer lifecycle on Vault leases, not on "short-TTL = no revocation."** Because env-ctl bearers are long (≤24h) relative to Fulcio/SPIRE, ship an explicit revoke that (a) invalidates immediately and (b) blocks further USB-gated renewal — exactly Vault's revoke-vs-expire semantics. Keep the resolve-once `UnlockContext` so revocation can't be raced. ([Vault lease](https://developer.hashicorp.com/vault/docs/concepts/lease))
3. **Renew on a fraction of TTL, not at the edge.** Adopt SPIRE-style proactive renewal (renew well before expiry; SPIRE uses 50%). For a 24h bearer, gate renewal behind USB presence and refresh at, e.g., 50–75% of TTL so a transient USB-absent window doesn't drop the agent. ([SPIRE](https://github.com/spiffe/spire/issues/4268))
4. **Make the C-deps ruling explicit and document it (this is the open OI-1 backend question).** If "no C deps" is a hard constraint, you cannot use mainline libSQL — it bundles `sqlite3.c`. Use **Turso** (pure Rust, native per-page encryption, ~6%/14% overhead) instead, and note env-ctl already does app-layer XChaCha20-Poly1305 so Turso's native encryption can be belt-and-suspenders or omitted. If C is acceptable, libSQL's maturity is a fine default. ([libSQL](https://github.com/tursodatabase/libsql), [Turso](https://turso.tech/blog/introducing-fast-native-encryption-in-turso-database))
5. **Keep the local-CA private key in the same USB-gated keyslot tier as the DEK.** The MITM CA key is as sensitive as the credential store (see Infisical/1Password threat model); protect it with the same argon2id keyslot + USB-UUID gating, never plaintext on disk. ([Infisical](https://infisical.com/blog/agent-vault-the-open-source-credential-proxy-and-vault-for-agents))
6. **Adopt an append-only audit log à la Rekor / Vault audit backend.** Record per-bearer issuance/renewal/revocation and per-relay credential injection (which credential, when, which peer uid/pid, under what authority). 1Password's "Audit" pillar and Rekor's witnessed append-only log are the reference shapes. ([1Password](https://1password.com/press/2026/mar/1password-unified-access), [Rekor v2](https://blog.sigstore.dev/rekor-v2-ga/))
7. **Use the real, current secrets-sprawl numbers in any positioning copy.** 29M secrets on public GitHub in 2025 (+34% YoY), 81% surge in AI-service leaks, ~64% of 2022-valid secrets still valid in 2026. Drop the unverified "53% in 1 week" stat. ([GitGuardian 2026](https://blog.gitguardian.com/the-state-of-secrets-sprawl-2026/))
8. **`SO_PEERCRED` peer scoping is correct — keep uid/pid binding on the UDS control plane.** This is the documented gRPC-UDS pattern; env-ctl's peer-bound bearer (resolve-once) exceeds the typical mint-time-only check. ([gRPC #15511](https://github.com/grpc/grpc/issues/15511))

## Open questions

1. **OI-1 / backend C-deps ruling (BLOCKER for store choice).** Is "no C dependencies" a hard constraint? If yes → Turso (pure Rust); if no → libSQL is fine. Mainline libSQL definitively bundles C `sqlite3.c`. ([libSQL](https://github.com/tursodatabase/libsql))
2. **Does env-ctl rotate the *injected upstream* secrets?** The relay model leaves the real upstream credential long-lived and shared. Without rotation, the broker reduces *exposure* but not *blast radius if the store is breached*. Define a rotation cadence/owner.
3. **Audience granularity.** Per-upstream-host, per-API-path, or per-method audience binding? RFC 9068 supports resource indicators; pick a granularity that matches your relay routing table.
4. **Rekor v2 client cutover timing** — if env-ctl ever logs to Sigstore or consumes its transparency log, confirm whether the SigningConfig has actually distributed the v2 URL yet (was pending as of Oct 2025, planned end-2025/early-2026; not re-confirmed live for June 2026). ([Rekor v2](https://blog.sigstore.dev/rekor-v2-ga/))
5. **Doppler / RFC 9106 / libsodium / gRPC-#15511 citations were carried over, not re-fetched this pass.** Re-verify before publishing externally if exactness matters (all are stable, low-risk).
6. **1Password runtime scoped credentials** are "later this year" — if env-ctl wants to compare against that feature, track whether it ships before env-ctl's own launch. ([1Password](https://1password.com/press/2026/mar/1password-unified-access))
