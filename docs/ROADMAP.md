# envctl — Roadmap

Six phases. **Phase 0 (this scaffold)** compiles green and ships a working, read-only `auto-detect`; the safety-critical guard engine is implemented fail-closed up front (not deferred).


## Phase 0 — Scaffold that compiles — A green Cargo workspace with the full type skeleton and the event/runner seams, no real behavior yet. Establishes the shared-engine boundary so CLI and GUI can never diverge.

- Cargo workspace with three members (envctl-engine lib, envctl CLI bin, envctl-gui eframe bin) and shared [workspace.dependencies]; cargo build green on stable Rust
- engine: component.rs (Component, Hook, Guard, Phase, HookRunner), model.rs (Registry, RunPlan, RunSummary, OpResult/OpStatus, Wiring), report.rs (EnvReport + sub-structs), event.rs (Event, EventSink, Stream, Telemetry), error.rs (EngineError + run_phase signature) — all serde-derived, compiling, with unimplemented!()/todo bodies
- exec.rs DryRunner + RecordingRunner stubs; ProcessHookRunner signature only
- CLI: clap parses all five verbs + --dry-run/--apply/--json/--only and prints a 'not yet implemented' line; exit codes wired to RunSummary::ok()
- GUI: eframe app opens, five empty screen tabs, spawns the worker thread + two channels, drain_events()/request_repaint() loop present (renders placeholders)
- ARCHITECTURE.md committed; one example manifest (manifests/bun.toml) deserializes in a unit test

## Phase 1 — auto-detect (read-only) — Make the whole picture observable without changing anything. The sensing substrate the other verbs consume, and the thing testable on the live dual-5090 box today.

- Registry::load — read every *.toml under manifests/ + ~/.config/envctl/components.d/, dedup by id, Kahn topo-sort with declaration-order tie-break, cycle -> EngineError
- Probes: host (os-release, kernel, sysinfo cpu/mem/disks, secure_boot, uefi), block_uuids (lsblk -J), wiring_state (bashrc markers, PATH membership, autostart, CDI, nix substituters), tool_versions (which + --version with timeout)
- GPU three-tier cascade: Tier 0 PCI floor (/sys + lspci), Tier 2 nvidia-smi CSV, /proc/driver/nvidia parse for driver_loaded/open_kernel_module; Tier 1 NVML behind optional gpu-nvml feature
- Engine::detect -> EnvReport with per-component detect/verify, ProbeWarnings (best-effort), preconditions (SecureBoot off, reboot-needed)
- Drift function: pure diff(EnvReport, manifest desired-state) -> DriftItems with suggested_verb + manifest-driven severity, unit-tested against a recorded EnvReport fixture
- CLI: envctl auto-detect [--json/-v/--gpu] prints the status table + drift; exit 0 always
- GUI Dashboard + Components grid render the live EnvReport read-only; runs on the worker, never blocks the UI

## Phase 2 — install (additive, idempotent, streaming) — Bring components to present+verified in dependency order by wrapping the wizard's bash verbatim, with idempotent wiring and live streamed logs.

- ProcessHookRunner: spawn Hook (Command/Script bash -lc/ShippedScript), pipe + line-stream stdout/stderr as Event::Log, classify exit -> OpResult; needs_sudo handling; tee to ~/.local/state/envctl/envctl.log
- run_phase() best-effort wrapper: gpu_required skip, NoHook, catch_unwind, sink.timed(); RunSummary.failed / skipped_blocked rollup mirroring fail[]
- Wiring::apply(): guarded BEGIN/END shell_rc blocks (grep-guard), PATH-append-if-absent, .desktop (incl. one_shot), update-alternatives — all idempotent check-then-apply
- Full base manifest authored: all 32 leaf + 6 group/meta components from the catalog, hooks wrapping the exact wizard invocations; reboot_gated_verify + version_lock set; boot-repair UUIDs in machine maps
- Forward traversal: skip-if-detected, SkippedBlocked on failed/missing dep, RebootRequired (not Failed) for reboot-gated verifies
- CLI: envctl install [COMPONENT...] with --dry-run/--only/--skip/--no-gpu/--force/--fail-fast/--json
- GUI: per-row Install + 'install all missing', Live Logs streaming, busy spinners, health pill from RunSummary

## Phase 3 — reset + auto-fix (destructive, full guard discipline) — Make the environment reversible and repairable under ubuntu-boot-repair.sh's safety model: dry-run default, resolve+re-verify, refuse on ambiguity, back up before clobber, never touch /home.

- RunContext: resolve live-device identities + GPU presence ONCE per run (no TOCTOU); guard.rs evaluates UuidResolves/NotLiveDevice/NotMounted/PathExists/HookSucceeds -> OpStatus::Refused + GuardRefused event
- Wiring::revert(): timestamped backup (.bak.$(date +%s)) then excise ONLY the owned marker block; symlink/unit deletion tolerant of missing targets
- reset: reverse topo order, --dry-run default/--apply, --cascade/refuse-on-dependents, --all + --confirm, --keep-config/--purge; re-detect to confirm ABSENT
- auto-fix: act only on BROKEN/PARTIAL, minimal corrective wiring/fix-hook actions, atomic backup->apply->verify with revert-on-verify-failure; --dry-run DEFAULT/--apply, --only-wiring
- Boot-repair delegation: wrap ubuntu-boot-repair.sh diagnose/repair-dev/rename-pro/finalize as ShippedScript components gated behind --allow-boot; UUIDs/bl-ids from manifest machine config
- CLI exit codes reflect refused/failed; GUI confirmation modals naming exactly what is unwired/backed up before any destructive op

## Phase 4 — add-repo (build-from-source + wire-in) — The extensibility verb: turn an arbitrary upstream repo into a first-class, idempotent, removable component registered as a drop-in.

- 9-stage pipeline in addrepo.rs: acquire (clone/fetch, record resolved SHA, snapshot local paths), detect build system (nix flake>cargo>meson>cmake>make>python/uv>bun/npm), best-effort dep resolution, build (kit env sourced, SHA+flags-keyed cache), locate+classify artifacts, install (symlink-default into ~/.local, backup-before-clobber), wire-in (guarded PATH/completions/desktop/systemd --user)
- Component synthesis + atomic drop-in write to ~/.config/envctl/components.d/<slug>.toml (temp+rename, provenance recorded)
- Refuse-on-ambiguity edges: unknown build system w/o --build-cmd, zero artifacts, name collision w/ built-in
- CLI: envctl add-repo <url|path> --name --ref/--build-cmd/--prefix/--bin/--depends/--dry-run/--no-wire/--force/--json
- GUI Add Repo screen: form -> AddRepo{dry_run} dispatch, Validate (detect-only), auto-switch to Live Logs, inline validation errors
- End-to-end: add-repo a real cargo --git tool, see it appear in auto-detect and be managed by install/reset/auto-fix

## Phase 5 — GUI polish + telemetry + hardening — Finish the dashboard experience and lock in the no-block-UI + safety guarantees; make it pleasant for daily use on the box.

- Live telemetry: dedicated ~1s sampler thread emitting Event::Telemetry (nvidia-smi CSV default, NVML when gpu-nvml built), cadence backoff off-Dashboard, pause when window unfocused; painter sparklines (~120-sample ring) for GPU util/temp + CPU
- DriverNotActive yellow 'REBOOT to load nvidia-open 610' card pre-reboot (graceful, never panics)
- Settings/Manifest screen: read-only hook viewer + global options (telemetry interval slider, log-cap, dry-run-default checkbox ON, confirm-destructive), reload-manifest
- Live Logs: bounded ring (~8k), level/component filters, pause-autoscroll, copy/save, sticky active-step header
- NDJSON --json event stream for CLI scripting; replayable on-disk run log
- Tests: RecordingRunner-driven golden runs, drift fixtures, wiring apply/revert round-trip, guard refusal cases; cargo build green with and without the gpu-nvml feature on stable Rust


## Key decisions

- **Components are data deserialized from TOML (one Component struct, five optional Hook phases), not a Rust trait implemented per tool. The only behavioral trait kept is HookRunner.** — Adding a tool becomes adding a TOML file, never recompiling. It directly satisfies the brief's declarative-manifest requirement and lets the engine WRAP the wizard's proven bash verbatim (Hook::Script / Hook::ShippedScript) instead of rewriting it. HookRunner is the one seam needed so the GUI/tests can inject dry-run/recording runners.
- **A failing hook is NOT a Rust Err. Per-phase execution is best-effort: a nonzero exit becomes OpStatus::Failed in an OpResult, recorded in RunSummary.failed, and the run continues. Engine::run returns Err only for setup-time problems (manifest parse, dependency cycle, unknown component) via typed thiserror EngineError; a hook spawn is wrapped in catch_unwind.** — This is yazelix-setup.sh's run() wrapper (lines 27-32, fail+=()) expressed in Rust: never abort the batch on one failure, surface a final 'items needing attention' roster. The two-layer thiserror (typed startup errors) + anyhow (leaf I/O) split is the mainstream idiom and keeps deps minimal.
- **Destructive phases (reset=Remove, auto-fix=Fix, wrapped boot-repair) are dry-run by default and require --apply; reset --all needs an extra --confirm. install stays additive (acts by default). There is NO --force that bypasses the resolve/re-verify/refuse-on-ambiguity guards.** — Inverts the default exactly per ubuntu-boot-repair.sh (read-only/dry-run unless an explicit write subcommand) while keeping install's act-by-default to match the wizard. Making the guards unconditional preserves the boot script's refuse-on-ambiguity guarantee.
- **Guards are a declarative enum (UuidResolves, NotLiveDevice, NotMounted, PathExists, HookSucceeds) checked before any destructive phase; a failure yields OpStatus::Refused (safe abort + red GuardRefused event), never Failed/panic. Machine-specific UUIDs/bootloader-ids live in each component's manifest 'machine' map, not as literals in engine code.** — Models ubuntu-boot-repair.sh's resolve_verified()/die() chain (lines 72-126) as data. The catalog flagged that the four boot-repair subcommands hardcode machine-specific UUIDs (DEV_ROOT_UUID, DEV_ESP_UUID, DEV_HOME_UUID, LIVE_ROOT_UUID) and bl-ids — those MUST be config, not shared-code constants, so a guard references them by key.
- **Wiring is a first-class declarative type (path_entries, shell_rc with BEGIN/END markers, desktop_entries with one_shot, systemd_user, alternatives). apply() is idempotent/grep-guarded; revert() excises ONLY the owned marker block after a timestamped backup.** — Mirrors the wizard's guarded ~/.bashrc blocks (cuda env, yazelix auto-enter), one-shot self-disabling gpu-verify .desktop, PATH appends, and ghostty update-alternatives. Excising only the marked block is the anti-clobber guarantee — unrelated user bashrc content is never lost, matching boot-repair's back-up-before-clobber discipline.
- **Streaming uses dependency-free std::sync::mpsc (unbounded, single-consumer); crossbeam-channel is documented as a one-type drop-in only if multi-consumer/select is ever needed. The engine never prints — it emits Events; the CLI drains on the main thread, the GUI drains via try_recv in update().** — Resolves the conflict between the engine output (std mpsc) and stray GUI wording: std mpsc is sufficient for the single-consumer CLI/GUI case and keeps the dependency count minimal as the brief demands. Unbounded = lossless under bursty apt/nix output; the engine-never-prints rule guarantees identical CLI and GUI behavior.
- **GPU detection is a three-tier cascade (Tier 0 PCI floor always-first/never-fails → Tier 1 NVML preferred → Tier 2 nvidia-smi CSV). nvidia-smi CSV is the always-available DEFAULT (zero extra deps, exactly what the kit does); NVML is an optional 'gpu-nvml' cargo feature.** — Reconciles the engine/GUI outputs (parse nvidia-smi CSV) with the detection output (prefer nvml-wrapper) against the 'few mainstream deps' constraint. The PCI floor gives an authoritative GPU count (2) for Xid/dropped-card detection and works on the driverless first boot; NVML upgrades to fork-free typed FFI only when present, degrading gracefully so init() failure is never a panic.
- **The richer EnvReport from the detection design (HostInfo, GpuReport, NvidiaStackReport, ComponentStatus, drift, preconditions, ProbeWarnings) is canonical; the engine's leaner EnvReport/ComponentState folds into it. auto-detect is read-only, never writes, and is the sensing substrate the other four verbs consume.** — The two outputs conflicted on EnvReport shape; the detection schema is the superset and is serde-serializable for CLI --json and GUI rendering without re-probing. Folding ComponentState into ComponentStatus keeps one type. Absence as Option/enum (never error) matches the wizard's never-abort ethos and makes the design testable on the live box today.
- **Reboot-gated verifies (nvidia-open, cuda-toolkit driver parts, pytorch-venv torch.cuda, cuda-oxide runtime) carry a reboot_gated_verify flag and report OpStatus::RebootRequired / a yellow 'REBOOT to load nvidia-open 610' card, not Failed. Drift severity is manifest-driven: version_lock=Critical, unlocked=Warn.** — The wizard explicitly notes the driver only goes live after reboot (lines 184-188, 246-248); marking these Failed pre-reboot would poison the best-effort roster. version_lock controls drift severity so the live 595.71.05-vs-desired-610 driver mismatch is Warn (intentionally unlocked per HANDOFF Decision #5) while cuda-toolkit-13-3 is Critical — letting engine/CLI/GUI rank identically.
- **Three crates: envctl-engine (lib), envctl (CLI bin), envctl-gui (eframe bin). add-repo writes synthesized components to ~/.config/envctl/components.d/*.toml drop-ins merged at load; builds install user-scope under ~/.local + repos cached at ~/.local/share/envctl/repos; logs tee to ~/.local/state/envctl/envctl.log.** — Resolves the crate-naming conflict (envctl-engine/envctl/envctl-gui chosen over envctl-core/envctl-cli) and the add-repo prefix conflict (canonical XDG layout). Drop-in merging means add-repo never edits a shared file in place, so clean removal = delete the drop-in. All behavior in the lib guarantees CLI and GUI cannot diverge; the on-disk log is the direct analogue of ~/yazelix-setup.log.
- **GUI is eframe/egui immediate-mode with one long-lived serial worker thread, two mpsc channels, busy as a HashSet in the App, repaints driven by ctx.request_repaint() after each event, telemetry on its own ~1s cadence. No Mutex on the hot path.** — Immediate mode means the UI is a pure function of the latest snapshot — no view/state drift, trivial worker-driven repaints, ~0% CPU at rest, and egui's built-ins cover the whole need with zero extra deps (the brief's NO-web / few-deps constraints). A single serial worker ensures two sudo/apt/nix destructive ops can never race; a separate sampler ensures a 10-minute CUDA build never freezes the GPU gauges.
- **Dependency order is a deterministic Kahn topological sort with ties broken by manifest declaration order; cycles are a hard load-time error. install/add-repo/auto-fix traverse forward (SkippedBlocked if a dep failed), reset traverses reversed (refuse-on-dependents unless --cascade), and gpu_required is an orthogonal lspci runtime gate.** — Deterministic ties make the built-ins reproduce the wizard's exact proven sequence (NerdFonts->Bun->AI CLIs->rustup->GPU subgraph->Nix->home-manager->yazelix). Refusing cycles at load is refuse-on-ambiguity applied to config. SkippedBlocked encodes the wizard's reason for front-loading Nix before home-manager/yazelix; the orthogonal GPU gate matches the section-3b lspci guard without polluting the dep graph.
