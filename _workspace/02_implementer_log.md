New `EngineCommand` variants (`command.rs`) + event-loop wiring:
- `Dashboard { start, meta_file, spec }`
- `DeployDashboard { start, meta_file, spec, dry_run, force }`
Both routed in `run_event_loop` to the identical engine methods the CLI calls (parity proof below).

`parse_workspace`, `render`, `deploy`, `layout_target`.

## KDL syntax (VERIFIED against zellij layout docs via WebFetch)
Top-level `layout { ... }`; `tab name="..." [focus=true]`; `pane name="..." cwd="..." command="..." { args "a" "b" }`.
Args MUST be a child node inside the pane braces (not inline) — confirmed and implemented.
The mission-control overview tab is emitted first with `focus=true`.

Golden KDL sample (render of the fixture, cwd root `/work`):
```kdl
// Generated by envctl dashboard — do not edit by hand.
// envctl-owned: regenerate with `envctl dashboard --deploy --apply`.
layout {
    tab name="mission-control" focus=true {
        pane name="overview" cwd="/work" command="envctl-dashboard-pane" {
            args "overview"
        }
    }
    tab name="meta-core" {
        pane name="meta_cli" cwd="/work/meta_cli" command="envctl-dashboard-pane" {
            args "meta_cli"
        }
        ...
    }
    tab name="tools/env" { pane name="envctl" cwd="/work/envctl" ... }
    tab name="ops" { ... }
    tab name="mcp" { ... }
}
```
The `// Generated by envctl dashboard` comment doubles as the deploy ownership marker.

## Handoff notes (for the invariant-guardian)
- **Fail-closed deploy lives in `dashboard::deploy(plan, dry_run, force)`** (engine), NOT in
  `wiring.rs`. It mirrors wiring discipline: dry-run default, backup-then-write
  (`.bak.<nanos>`), and REFUSES to overwrite a file lacking the `// Generated by envctl dashboard`
  owner marker unless `force`. Refusal path is unit-tested
  (`deploy_refuses_foreign_file_without_force`) AND integration-tested
  (`deploy_dry_run_writes_no_file` asserts NO file appears). Verify both.
- **CLI default is read-only** (`envctl dashboard` prints KDL to stdout). Mutation requires
  `--deploy --apply`; `--deploy` alone is dry-run. The CLI maps `--apply`→`dry_run=false`.
- **Revert-on-reset is NOT wired this pass** — that is the Pass-2 manifest component (`install/fix
  → envctl dashboard --deploy --apply`; remove reverts). The deploy already backs up, so reset
  modeling has the backup to restore. Plan step 10 (component) + 12 (lock) are Pass 2.
- **YAML reader uses BTreeMap** (sorted key order) as the canonical "declaration order" — YAML
  maps don't preserve insertion order via serde_yaml. The meta-core tab + golden test depend on
  this sorted order; flag if the architect intended true source-file order (would need a
  preserve-order YAML mode).
- **The launcher `command=` in KDL is `envctl-dashboard-pane`** (bare, PATH-resolved). Pass-2
  install must put it on PATH (symlink into `~/.local/bin`, like other assets).
